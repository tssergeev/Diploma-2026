# Центральный сервер-медиатор

Этот проект реализует центральный сервер для прототипа мультибазовой системы интеграции показателей здоровья пользователей спортивных организаций.

Сервер работает как **медиатор**: он подключается к трём независимым клиентам-источникам, получает от них данные через REST API, приводит разные локальные структуры к единой глобальной модели и предоставляет поиск по интегрированным профилям.

## Роль проекта в архитектуре

Вся связка прототипа состоит из четырёх локальных приложений:

| Компонент | Порт | Роль |
|---|---:|---|
| `fitness_club_client` | `8001` | Первый источник: фитнес-клуб, реляционная структура |
| `sports_school_client` | `8002` | Второй источник: спортивная школа, собственная локальная схема |
| `fitness_app_client` | `8003` | Третий источник: фитнес-приложение, JSON-документы |
| `mediator_server` | `8000` | Центральный сервер-медиатор |

Медиатор не должен рассматриваться как простое централизованное хранилище. В прототипе он создаёт локальное интегрированное представление для демонстрации работы, но логически выполняет функции:

- регистрации источников данных;
- проверки доступности источников;
- вызова обёрток для каждого типа клиента;
- преобразования локальных схем в глобальную схему;
- дедупликации пользователей;
- хранения глобальных профилей;
- учёта согласий и статуса «открыт к предложениям»;
- поиска пользователей по критериям;
- журналирования действий.

## Быстрый запуск сервера

```bash
cd mediator_server
python -m venv venv
source venv/bin/activate
# Windows: venv\Scripts\activate

pip install -r requirements.txt
python manage.py migrate
python manage.py seed_sources
python manage.py runserver 127.0.0.1:8000
```

Проверка:

```text
http://127.0.0.1:8000/api/status/
```

## Полный запуск всей системы

Откройте четыре отдельных терминала.

### Терминал 1 — первый клиент

```bash
cd fitness_club_client
source venv/bin/activate
python manage.py runserver 127.0.0.1:8001
```

### Терминал 2 — второй клиент

```bash
cd sports_school_client
source venv/bin/activate
python manage.py runserver 127.0.0.1:8002
```

### Терминал 3 — третий клиент

```bash
cd fitness_app_client
source venv/bin/activate
python manage.py runserver 127.0.0.1:8003
```

### Терминал 4 — медиатор

```bash
cd mediator_server
source venv/bin/activate
python manage.py runserver 127.0.0.1:8000
```

Если клиенты ещё не заполнены тестовыми данными, в каждом клиенте нужно предварительно выполнить:

```bash
python manage.py migrate
python manage.py seed_demo_data
```

Для сервера-медиатора:

```bash
python manage.py migrate
python manage.py seed_sources
```

## Синхронизация данных

Когда три клиента запущены, выполните синхронизацию в проекте медиатора:

```bash
python manage.py sync_sources
```

Или через HTTP:

```bash
curl -X POST http://127.0.0.1:8000/api/sync/ -H "Content-Type: application/json" -d "{}"
```

Синхронизация одного источника:

```bash
python manage.py sync_sources --source fitness_club_client_1
```

или:

```bash
curl -X POST http://127.0.0.1:8000/api/sync/ \
  -H "Content-Type: application/json" \
  -d "{\"source_id\": \"fitness_app_client_3\"}"
```

## Основные API-эндпоинты

### Статус медиатора

```text
GET /api/status/
```

### Список источников

```text
GET /api/sources/
```

### Проверка доступности клиентов

```text
GET /api/sources/check/
```

### Регистрация или обновление источника

```text
POST /api/sources/
Content-Type: application/json

{
  "source_id": "custom_source",
  "name": "Дополнительный источник",
  "source_type": "generic_rest",
  "base_url": "http://127.0.0.1:8010/api/",
  "users_endpoint": "users/?include_nested=true"
}
```

### Синхронизация

```text
POST /api/sync/
Content-Type: application/json

{}
```

### Список глобальных профилей

```text
GET /api/users/
```

С вложенными активностями, показателями здоровья и согласиями:

```text
GET /api/users/?include_nested=true
```

### Поиск пользователей

```text
GET /api/search/users/?open_to_offers=true
```

Примеры фильтрации:

```text
GET /api/search/users/?gender=female&min_age=18&max_age=35&open_to_offers=true
GET /api/search/users/?city=Москва&sport_type=fitness&open_to_offers=true
GET /api/search/users/?metric_type=hrv&min_metric_value=50&open_to_offers=true
GET /api/search/users/?activity_type=cardio&min_activities=2&period_days=90
GET /api/search/users/?source_id=fitness_app_client_3&include_nested=true
```

### Просмотр одного профиля

```text
GET /api/users/<global_id>/
```

### Создание предложения от тренера

```text
POST /api/offers/
Content-Type: application/json

{
  "global_id": "UUID_ПОЛЬЗОВАТЕЛЯ",
  "trainer_name": "Петров Иван",
  "organization_name": "FitEnergy",
  "message": "Здравствуйте! Готов предложить программу персональных тренировок."
}
```

Предложение можно создать только для пользователя, у которого активен статус `open_to_offers=true`.

### Журнал аудита

```text
GET /api/audit/
GET /api/audit/?limit=100
```

## Глобальная модель медиатора

В проекте используются следующие основные сущности:

### `DataSource`

Хранит сведения о подключённых источниках:

- код источника;
- тип источника;
- базовый URL;
- эндпоинты статуса, схемы и пользователей;
- mapping-конфигурацию;
- дату последней синхронизации;
- последнюю ошибку подключения.

### `GlobalUserProfile`

Унифицированный профиль пользователя:

- глобальный UUID;
- ФИО;
- дата рождения или возраст;
- пол;
- контакты;
- город;
- вид спорта;
- уровень подготовки;
- организация;
- статус открытости к предложениям;
- объединённые категории видимости данных.

### `SourceUserLink`

Связывает глобальный профиль с локальной записью конкретного источника.

Например:

```text
fitness_club_client_1: member_001 -> global UUID
fitness_app_client_3: usr_10458 -> тот же global UUID
```

### `ActivityRecord`

Унифицированная активность:

- дата;
- тип активности;
- длительность;
- интенсивность;
- калории;
- источник.

### `HealthMetricRecord`

Унифицированный показатель здоровья:

- тип показателя;
- значение;
- единица измерения;
- дата измерения;
- источник.

### `ConsentRecord`

Согласие пользователя из конкретного источника:

- открыт ли пользователь к предложениям;
- статус согласия;
- видимые категории данных;
- разрешённые типы организаций;
- дата выдачи и дата истечения.

### `Offer`

Предложение от тренера пользователю.

### `AuditLog`

Журнал действий медиатора.

## Как работает дедупликация

При синхронизации медиатор пытается определить, относится ли новая локальная запись к уже существующему глобальному профилю.

Порядок поиска совпадений:

1. уже существующая связка `source + local_user_id`;
2. совпадение по email;
3. совпадение по телефону;
4. совпадение по fingerprint;
5. совпадение по ФИО и дате рождения;
6. совпадение по ФИО и возрасту.

Если совпадение не найдено, создаётся новый `GlobalUserProfile`.

## Как работает слой обёрток

Для каждого источника используется отдельная обёртка:

| Источник | Класс обёртки |
|---|---|
| Фитнес-клуб | `FitnessClubWrapper` |
| Спортивная школа | `SportsSchoolWrapper` |
| Фитнес-приложение | `FitnessAppWrapper` |
| Произвольный REST-источник | `BaseRestWrapper` |

Обёртка отвечает за получение данных от конкретного клиента. После этого модуль `normalization.py` приводит данные к единому виду.

## Как очистить интегрированные данные

```bash
python manage.py clear_integrated_data
```

Команда удаляет глобальные профили, активности, показатели, согласия, предложения и аудит, но не удаляет настройки источников.

## Как создать администратора

```bash
python manage.py createsuperuser
```

Админ-панель:

```text
http://127.0.0.1:8000/admin/
```

## Рекомендуемый порядок демонстрации

1. Запустить три клиента на портах `8001`, `8002`, `8003`.
2. Запустить сервер-медиатор на порту `8000`.
3. Открыть `/api/sources/check/` и показать, что источники доступны.
4. Выполнить `/api/sync/` или `python manage.py sync_sources`.
5. Открыть `/api/users/?include_nested=true`.
6. Показать поиск по критериям через `/api/search/users/`.
7. Создать предложение тренера через `/api/offers/`.
8. Показать журнал аудита через `/api/audit/`.

Такой сценарий демонстрирует все ключевые элементы прототипа: гетерогенные источники, медиатор, обёртки, mapping, нормализацию, дедупликацию, согласия, поиск и контролируемое взаимодействие между тренером и пользователем.

## HTML-панель статистики

В обновлённой версии сервер-медиатор содержит не только REST API, но и локальную HTML-панель для демонстрации работы прототипа.

Основные страницы:

| Страница | Адрес | Назначение |
|---|---|---|
| Сводная статистика | `http://127.0.0.1:8000/` | Общие показатели по источникам, профилям, активностям, метрикам и журналу действий |
| Статистика по клиентам | `http://127.0.0.1:8000/clients/` | Сравнение трёх клиентских источников и данных, полученных от каждого клиента |
| Карточка клиента | `http://127.0.0.1:8000/clients/<source_id>/` | Детальная статистика конкретного источника: профили, активности, показатели и параметры подключения |
| Персональная статистика | `http://127.0.0.1:8000/people/` | Поиск интегрированных пользователей и просмотр краткой статистики по каждому человеку |
| Карточка человека | `http://127.0.0.1:8000/people/<global_id>/` | Детальная персональная статистика: источники, активности, показатели здоровья, согласия и предложения |

На страницах панели доступны кнопки синхронизации. Перед синхронизацией должны быть запущены три клиента на портах `8001`, `8002` и `8003`.

Порядок запуска остаётся прежним:

```bash
python manage.py migrate
python manage.py seed_sources
python manage.py runserver 127.0.0.1:8000
```

После запуска сервера откройте:

```text
http://127.0.0.1:8000/
```

REST API сохранён без изменений и по-прежнему доступен по адресам `/api/status/`, `/api/sources/`, `/api/users/`, `/api/search/users/`, `/api/sync/`.
