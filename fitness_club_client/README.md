# Первый клиент прототипа: источник данных фитнес-клуба

Этот проект реализует первый локальный клиент для прототипа мультибазовой системы. Клиент имитирует независимый источник данных фитнес-клуба и отдаёт сведения серверу-медиатору через REST-подобные HTTP-эндпоинты.

В проект добавлена локальная панель управления для просмотра записей и добавления новых данных.

## Что реализовано

В клиенте есть локальная реляционная структура данных:

- `Organization` — фитнес-клуб или филиал;
- `Member` — пользователь / член клуба;
- `Activity` — посещение или тренировка;
- `HealthMetric` — показатель здоровья или активности;
- `Consent` — согласие пользователя и статус открытости к предложениям.

Также добавлены страницы:

- `/` — главная панель со счётчиками и последними записями;
- `/organizations/` — просмотр и добавление организаций;
- `/members/` — просмотр и добавление пользователей;
- `/activities/` — просмотр и добавление активностей;
- `/metrics/` — просмотр и добавление показателей здоровья;
- `/consents/` — просмотр и добавление согласий.

API-эндпоинты остались прежними и доступны в разделе `/api/`.

## Установка и запуск

```bash
cd fitness_club_client
python -m venv venv
```

### Windows

```bash
venv\Scripts\activate
pip install -r requirements.txt
python manage.py migrate
python manage.py seed_demo_data
python manage.py runserver 127.0.0.1:8001
```

### Linux / macOS

```bash
source venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py seed_demo_data
python manage.py runserver 127.0.0.1:8001
```

После запуска клиент будет доступен по адресу:

```text
http://127.0.0.1:8001/
```

## Работа через HTML-интерфейс

Откройте в браузере:

```text
http://127.0.0.1:8001/
```

На главной странице отображаются счётчики записей и последние добавленные данные. Навигация расположена в боковой панели.

Рекомендуемый порядок ручного добавления данных:

1. добавить организацию на странице `/organizations/`;
2. добавить пользователя на странице `/members/`;
3. добавить активности пользователя на странице `/activities/`;
4. добавить показатели здоровья на странице `/metrics/`;
5. добавить согласие пользователя на странице `/consents/`.

Для полей со списками в согласии используется JSON-массив. Пример для открытых категорий данных:

```json
["activity", "heart_rate", "hrv"]
```

Пример для разрешённых типов организаций:

```json
["fitness_club", "personal_trainer"]
```

## Основные API-эндпоинты

### Проверка состояния клиента

```text
GET /api/status/
```

Пример:

```bash
curl http://127.0.0.1:8001/api/status/
```

### Описание локальной схемы и mapping

```text
GET /api/schema/
```

Этот эндпоинт нужен серверу-медиатору, чтобы понять, какие локальные сущности есть у источника и как они сопоставляются с глобальной схемой.

### Получение пользователей

```text
GET /api/members/
```

Пример:

```bash
curl "http://127.0.0.1:8001/api/members/?open_to_offers=true&include_nested=true"
```

### Получение одного пользователя

```text
GET /api/members/<local_id>/
```

Пример:

```bash
curl http://127.0.0.1:8001/api/members/fc_1001/
```

### Получение активностей

```text
GET /api/activities/
```

Примеры:

```bash
curl http://127.0.0.1:8001/api/activities/
curl "http://127.0.0.1:8001/api/activities/?local_user_id=fc_1001"
curl "http://127.0.0.1:8001/api/activities/?activity_type=cardio"
```

### Получение показателей здоровья

```text
GET /api/metrics/
```

Примеры:

```bash
curl http://127.0.0.1:8001/api/metrics/
curl "http://127.0.0.1:8001/api/metrics/?metric_type=hrv"
```

### Получение согласий

```text
GET /api/consents/
```

Пример:

```bash
curl "http://127.0.0.1:8001/api/consents/?open_to_offers=true&status=active"
```

## Примеры поисковых запросов

Найти пользователей, открытых к предложениям:

```bash
curl "http://127.0.0.1:8001/api/members/?open_to_offers=true"
```

Найти женщин 25–35 лет, открытых к предложениям:

```bash
curl "http://127.0.0.1:8001/api/members/?gender=female&min_age=25&max_age=35&open_to_offers=true"
```

Найти пользователей с HRV выше 50 мс:

```bash
curl "http://127.0.0.1:8001/api/members/?metric_type=hrv&min_metric_value=50&open_to_offers=true"
```

Найти пользователей, которые посещали клуб минимум 3 раза за последние 90 дней:

```bash
curl "http://127.0.0.1:8001/api/members/?min_visits=3&period_days=90&open_to_offers=true"
```

## Как это будет использовать сервер-медиатор

Сервер-медиатор сможет работать с этим клиентом по такой логике:

1. запросить `/api/schema/` и получить описание структуры источника;
2. запросить `/api/members/`, `/api/activities/`, `/api/metrics/`, `/api/consents/`;
3. применить правила mapping;
4. привести данные к глобальной схеме;
5. объединить их с данными других клиентов;
6. выполнить поиск по критериям тренера.


## Смысл для дипломной работы

Этот клиент демонстрирует реляционный вариант источника данных фитнес-клуба. Он показывает, что локальный источник может иметь собственную структуру хранения, собственный интерфейс ввода данных и отдельный API для подключения к центральному серверу-медиатору.
